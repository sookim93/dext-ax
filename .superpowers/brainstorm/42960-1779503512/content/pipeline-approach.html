Zarvis 스킬 대시보드 — 전체 파이프라인 접근 방식

3가지 아키텍처 중 어떤 방향이 맞나요?

A

정적 웹앱 + JSON 데이터

구성: HTML/CSS/JS + skills.json

업데이트: JSON 교체만으로 반영 · GitHub Pages 배포 가능

수집: Python 스크립트가 Claude/ChatGPT 문서 크롤링 → JSON 생성

장점

  • 서버 불필요 · 배포 즉시
  • 유지보수 최소
  • 속도 빠름

단점

  • 실시간 업데이트 불가
  • 즐겨찾기 저장 어려움
B

Notion DB + 임베드 대시보드

구성: Notion이 마스터 데이터 → 웹 대시보드가 API 읽기

업데이트: Notion에서 직접 추가/수정 → 즉시 반영

수집: 스크립트가 Notion 페이지에 자동 추가

장점

  • 비개발자도 스킬 추가 가능
  • 팀 협업 용이

단점

  • Notion API 의존성
  • 로딩 느릴 수 있음
C

풀스택 웹앱 (Flask + SQLite + JS)

구성: 백엔드 API + DB + 프론트엔드

업데이트: 관리자 UI에서 직접 추가/수정/삭제

수집: 크론 잡 자동 크롤링 → DB 저장

장점

  • 검색·필터·즐겨찾기 등 풍부한 기능
  • 사용 통계 추적
  • 확장성 최고

단점

  • 개발·운영 비용 높음
  • 서버 필요

전체 파이프라인 3단계 (공통)

1단계 · 수집

Claude Code / ChatGPT / Gemini 공식 문서 + 커뮤니티 크롤링

2단계 · 대시보드

카테고리 탐색 · 검색 · 복사 · AI별 필터

3단계 · 업데이트

GitHub Actions cron · 변경 감지 · 알림
