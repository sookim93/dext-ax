# 덱스트 자동화 멀티에이전트 개발 시스템 설계서

## 1. 프로젝트 개요

덱스트 업무 자동화를 위한 4개 독립 모듈을, Claude Code 기반 멀티에이전트(오케스트레이터 + TF) 체제로 병렬 개발하고, 각 모듈은 직원 누구나 브라우저로 사용할 수 있는 독립 웹 애플리케이션으로 배포한다.

### 핵심 제약 조건

- **개발 인력**: 1인 개발자 + Claude Code (Max 요금제, API 비용 없음)
- **배포**: 로컬 서버 시작 → GitHub Actions/클라우드 이전
- **사용자**: 개발자가 아닌 다른 직원들, 수시로 변경될 수 있음
- **각 모듈의 AI 수준**: 모듈별로 개발하면서 결정

---

## 2. 시스템 구조

시스템은 두 개의 평면으로 구성된다.

### A) 개발 평면 — 오케스트레이터 + TF

개발자는 오케스트레이터(Claude Code 메인 세션)와만 소통한다. 오케스트레이터가 모듈별 TF(서브에이전트)를 worktree로 파견하여 병렬 개발을 수행한다.

```
[개발자]
 └── 오케스트레이터 (Claude Code 메인 세션)
      ├── TF1: 입찰공고 (서브에이전트, worktree 격리)
      ├── TF2: 수식추출 (서브에이전트, worktree 격리)
      ├── TF3: 견적서 (서브에이전트, worktree 격리)
      └── TF4: 결제정리 (서브에이전트, worktree 격리)
```

**오케스트레이터 역할:**

| 기능 | 설명 |
|------|------|
| 기획 | 모듈 스펙 정의 → 해당 TF에 전달 |
| 개발 지시 | "입찰공고 모듈에 이 기능 추가해" → TF 파견 |
| 병렬 개발 | 여러 TF 동시 파견 가능 |
| 진행 관리 | 전체 TF 상태 추적, 우선순위 조정 |
| 리뷰 | TF 결과물 검토 |
| 배포 | 완성된 모듈 배포 지시 |

**세션 간 컨텍스트 유지 규칙:**

- 오케스트레이터는 세션 시작 시 `docs/status.md`를 먼저 읽어 현재 진행 상황을 파악한다
- 각 TF는 작업 완료 시 `docs/status.md`를 자동 업데이트한다 (무엇을 했는지, 다음 할 일)
- 이 규칙은 오케스트레이터 CLAUDE.md와 각 TF의 AGENTS.md에 명시한다

**TF(서브에이전트) 역할:**

- 각 모듈 폴더의 AGENTS.md에 역할/규칙 정의
- 자기 폴더 안에서만 작업 (다른 TF와 충돌 없음)
- 설계/코딩/테스트를 독립적으로 수행
- 결과물은 Claude Code 없이 동작하는 독립 프로그램

**기술적 동작 방식:**

- 오케스트레이터는 Claude Code의 Agent 도구(worktree 격리 모드)로 TF를 파견
- 각 TF는 독립된 git worktree에서 작업하므로 다른 TF와 코드 충돌 없음
- TF의 AGENTS.md가 해당 서브에이전트의 컨텍스트(역할, 기술 스택, 규칙)를 정의
- 작업 완료 시 worktree의 변경사항이 메인 브랜치에 머지됨

### B) 운영 평면 — 4개 독립 웹 애플리케이션

각 TF의 결과물. Claude Code 없이 직원 누구나 브라우저로 사용한다.

---

## 3. 모듈별 결과물 정의

### 3-1. 입찰공고 (우선순위: 긴급)

**목적**: 나라장터(G2B) 입찰공고를 자동 모니터링하고, 필터링/정리/의사결정을 지원

**기능 흐름:**

```
[크론 자동실행] 나라장터 Open API 호출
       ↓
[1차 필터] 키워드 + 발주기관 유형(대학) + 금액 범위 + 커스텀 규칙
       ↓
[웹 대시보드] 필터 통과 공고 목록 표시
       ↓
[담당자 2차 확인] 참여/패스 결정
       ↓
[자동 정리] 관련 서류 다운로드 + 구글 스프레드시트 기입
```

**사용 방식:**
- 자동: 크론이 매일 체크 → 결과를 대시보드에 표시
- 수동: 대시보드에서 담당자가 참여/패스 버튼 클릭
- 서류 정리: 참여 결정 시 자동으로 서류 다운로드 + 시트 기입

**사용자:** 담당자 (수시로 변경 가능)

### 3-2. 수식추출 (우선순위: 파급력 큼)

**목적**: 대학 모집요강 PDF에서 수시/정시 점수 산출 공식(배점 비율, 반영 비율, 가중치 테이블 등)을 자동 추출

**"수식"이란:** 수학 수식이 아닌 입시 점수 산출 공식. 예: "학생부 40% + 수능 60%", 영역별 반영 비율표, 가산점 규칙 등. 모집요강 PDF 내 텍스트/표 형태로 존재.

**기능 흐름:**

```
[사용자] 모집요강 PDF 업로드
       ↓
[PDF 파싱] 텍스트 + 표 추출
       ↓
[공식 추출] 수시/정시 점수 산출 공식, 반영 비율, 가중치 파싱
       ↓
[결과 표시] 구조화된 공식 데이터 + 다운로드 옵션
```

**사용 방식:** 웹 앱에서 PDF 업로드 → 결과 확인/다운로드
**사용자:** 담당 직원 (변경 가능)

### 3-3. 견적서 (우선순위: 파급력 큼)

**목적**: 대학별 견적서를 작성하고 담당자 확인 후 메일로 자동 발송

**기능 흐름:**

```
[사용자] 대학, 금액, 담당자 메일 선택
       ↓
[견적서 자동 생성] 템플릿 기반 작성
       ↓
[미리보기] 담당자 확인
       ↓
[발송 버튼] 메일 자동 발송
```

**메일 발송:** 담당자 개인 Daum 회사 메일 계정에서 발송. SMTP 또는 Daum 메일 API 연동 필요. 발송 방식 상세는 TF3이 개발 시 결정.

**사용 방식:** 웹 앱 (모바일 반응형) — 이동 중에도 확인/발송 가능
**사용자:** 담당자 (변경 가능)

### 3-4. 결제정리 (우선순위: 후순위)

**목적**: 혜정팀장님이 수행하는 결제 정리 업무 자동화

**현재 상태:** 상세 워크플로우 미파악. 혜정팀장님 업무 프로세스 파악 후 별도 설계 필요.

**사용자:** 혜정팀장님 및 후임자

## 4. 기술 스택

### 원칙

공통 스택을 강제하지 않는다. 각 TF가 모듈 특성에 맞는 최적 스택을 자율 결정한다.

| 공통 사항 | 내용 |
|----------|------|
| 데이터 | 구글 스프레드시트 연동 (기존 워크플로우 유지) + 필요 시 로컬 DB |
| 배포 | 로컬 서버에서 시작 (각 앱 독립 포트), 이후 클라우드 이전 가능 |

### 모듈별 스택 방향 (TF가 최종 결정)

| 모듈 | 핵심 기술 요소 | 스택 방향 예시 |
|------|--------------|--------------|
| 입찰공고 | 나라장터 Open API (data.g2b.go.kr), 필터링, 스프레드시트 연동, 크론 | Python 중심 (API 호출/자동화 생태계 활용) |
| 수식추출 | PDF 파싱, 수식 인식 | Python 중심 (PDF/AI 라이브러리 풍부) |
| 견적서 | 모바일 반응형 UI, 메일 발송, 템플릿 | 풀스택 JS 또는 Python — 모바일 UX가 핵심 |
| 결제정리 | 업무 파악 후 결정 | TBD |

## 5. 프로젝트 폴더 구조

```
dext-ax/
├── CLAUDE.md                ← 오케스트레이터 설정 (TF 라우팅, 프로젝트 규칙)
├── docs/
│   └── status.md            ← 전체 TF 진행 현황 대시보드
├── projects/
│   ├── bidding/             ← TF1: 입찰공고 (독립 앱)
│   │   ├── AGENTS.md        ← TF1 서브에이전트 역할/규칙
│   │   ├── README.md
│   │   └── (내부 구조는 TF가 모듈 특성에 맞게 결정)
│   ├── formula/             ← TF2: 수식추출 (독립 앱)
│   │   ├── AGENTS.md
│   │   └── ...
│   ├── quotation/           ← TF3: 견적서 (독립 앱)
│   │   ├── AGENTS.md
│   │   └── ...
│   └── payment/             ← TF4: 결제정리 (독립 앱)
│       ├── AGENTS.md
│       └── ...
└── .github/
    └── workflows/           ← 입찰공고 자동 체크 등 스케줄 작업
```

각 `projects/xxx/`는 자기만의 의존성, README, 배포 방식을 가진 완전 독립 프로젝트.

## 6. 개발 순서

```
Phase 1: 기반 세팅
├── git 초기화 + monorepo 폴더 구조 생성
├── harness-100에서 2~3개 하네스 분석 → 작성 패턴 추출
├── 오케스트레이터 CLAUDE.md 작성
├── TF별 AGENTS.md 작성
└── docs/status.md 초기 생성 (전체 TF 진행 현황 템플릿)

Phase 2: 입찰공고 (긴급)
└── TF1 파견 → 나라장터 Open API + 1차 필터 + 웹 대시보드 + 크론 + 서류정리

Phase 3: 수식추출 & 견적서 (병렬 개발 — 파급력)
├── TF2 파견 → PDF 파싱 + 수식 추출 웹앱
└── TF3 파견 → 견적서 작성/발송 웹앱 (동시 진행)

Phase 4: 결제정리
└── TF4 파견 → 혜정팀장님 업무 파악 후 설계/개발
```

## 7. 미결정 사항

| 항목 | 현재 상태 | 결정 시점 |
|------|----------|----------|
| 각 모듈 AI 활용 수준 | 미정 | 모듈별 개발 시 |
| 결제정리 모듈 상세 | 업무 프로세스 미파악 | Phase 4 착수 시 |
| 인증 방식 | 모듈별 TF가 결정 (수식추출은 불필요, 견적서는 메일 연동이 인증 역할, 입찰공고는 의사결정 추적 필요 시 간단 인증) | 각 모듈 개발 시 |
| 클라우드 이전 시점 | 로컬 안정화 후 | 운영 중 판단 |
| 입찰공고 2차 확인 알림 채널 | 웹 대시보드 확정, 추가 알림(카톡 등) 미정 | Phase 2 개발 시 |
